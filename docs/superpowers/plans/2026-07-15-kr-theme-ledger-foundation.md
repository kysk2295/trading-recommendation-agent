# KR Theme Ledger Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans or superpowers:subagent-driven-development task by task. Follow test-driven development. This milestone is local-only and must not add KIS order/account APIs, live broker calls, an LLM network call, or a trading signal.

**Goal:** Create the private, append-only evidence foundation for the independent `kr_equities` theme opportunity vertical: raw catalyst BLOBs, per-cycle source coverage, and versioned stored theme classifications.

**Architecture:** A separate mode-600 SQLite ledger owns KR catalyst data. Deterministic catalyst identities deduplicate the same source record while a cycle-observation table preserves every collection cycle in which it was seen. Final cycle records require exact coverage for news, DART, KIS domestic ranking, and derived volume surge. Classification rows reference immutable catalyst IDs and preserve classifier, prompt, and run versions. A local manifest CLI exercises raw-first ingestion without choosing a production news provider or calling an LLM.

**Tech Stack:** Python 3.12, Pydantic 2, SQLite, Typer, pytest, Ruff, basedpyright, uv

---

## Safety And Scope Invariants

- `MarketId.KR_EQUITIES` remains shadow/research only with no legacy execution binding.
- New modules do not import KIS auth/client, Alpaca, broker mutation, account, position, or order code.
- Raw payloads stay in the private SQLite BLOB ledger and never appear in CLI output, reports, logs, cards, or committed fixtures except tiny synthetic payloads.
- `published_at` is the source-claimed time; `observed_at` is the local availability time. They are never substituted for each other.
- A cycle is final only when all four source coverage entries are present exactly once, including explicit failures.
- Same immutable identity and content is idempotent; changed content under the same identity fails closed.
- UPDATE and DELETE are blocked by SQLite triggers for every ledger table.
- Writer uses one nonblocking lease; readers use `mode=ro` and `PRAGMA query_only=ON`.
- No classification result is generated by this milestone. It only defines and stores results supplied by a later deterministic keyword or LLM classification adapter.

- recapturing the same source record in the same cycle is a no-op;
- seeing it in a later cycle adds an observation without duplicating the raw row;
- same source identity with changed metadata or payload raises an immutable conflict;
- an earlier claimed first observation after a row already exists raises a conflict;

Use a nonblocking `fcntl` writer lease and WAL. Insert raw catalyst and its observation in one transaction. Finalize a cycle only after querying exact association counts. Reader methods never render BLOB contents.
